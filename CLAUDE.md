# Doseify — Claude Code Project Guide

This file is persistent context for Claude Code. Read it on every session.

## What this is

A warm, travel-aware medication tracker for a single user managing CLL (Chronic Lymphocytic Leukemia) drugs. iPhone + Apple Watch. Local data only. The differentiator is intelligent timezone handling for travel.

**This app helps a real person manage real cancer medication.** Bias toward correctness over cleverness. When in doubt, prefer the safer path.

## Read these before writing code

1. `SPEC.md` — full product specification. Source of truth for features, data model, edge cases.
2. `README.md` — human-facing setup notes (not normally relevant to you).

If `SPEC.md` and this file ever conflict, **SPEC.md wins** for product behaviour; this file wins for tooling and conventions.

## Tech stack (fixed — do not substitute)

- **Swift 5.10+, SwiftUI** for all UI
- **SwiftData** for persistence (local only, no CloudKit)
- **UNUserNotificationCenter** for notifications, with categories and actions
- **WidgetKit** for home-screen and Lock Screen widgets, plus Watch complications
- **WatchConnectivity** for Watch ↔ phone sync
- **HealthKit** (write-only initially)
- **BackgroundTasks** (BGAppRefreshTask) for scoring rollover and missed-dose marking
- **PDFKit** for adherence report export
- **Min iOS 17 / watchOS 10**

No third-party dependencies in v1 unless absolutely required. If you think one is needed, ask first.

## Project generation

The project is generated from `project.yml` by `xcodegen` — never hand-edit `project.pbxproj`.
`project.yml` globs `path: Doseify`, so new files in that tree are picked up on the next
`xcodegen generate`. The `.xcodeproj` is gitignored, so regenerate after a fresh checkout or
whenever source files are added/removed/renamed.

```bash
xcodegen generate          # regenerate from project.yml
open Doseify.xcodeproj      # or `xed .`
```

If `xcodegen` is missing, install with `brew install xcodegen`. If it's unavailable and
cannot be installed, **stop and ask** — hand-writing `.xcodeproj` files is not allowed here.

## Build, run, test commands

This machine's `xcodebuild` is reached via the `DEVELOPER_DIR` override (the bare CLI
points at CommandLineTools, not full Xcode, and fails without it). No `sudo` needed.
Use an installed simulator name — `iPhone 17` exists here; `iPhone 15` does not.

```bash
# All xcodebuild invocations need this prefix (or `export` it once per shell):
export DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer

# Regenerate the Xcode project after adding/removing/renaming source files
# or editing project.yml (the .xcodeproj is gitignored, not committed).
xcodegen generate

# Build for simulator (run from repo root)
xcodebuild -project Doseify.xcodeproj -scheme Doseify -destination 'platform=iOS Simulator,name=iPhone 17' build

# Run unit tests
xcodebuild -project Doseify.xcodeproj -scheme Doseify -destination 'platform=iOS Simulator,name=iPhone 17' test

# Lint (if SwiftLint is set up)
swiftlint --strict
```

When running tests, the app launches in the test host and floods stderr with harmless
`CoreData: error: Sandbox access ... default.store` lines — filter with
`grep -vE "CoreData:|Doseify\[[0-9]"` to read the actual Swift Testing results.

Prefer `xcodebuild` over opening Xcode for verification — it gives parseable output and works headlessly.

## Conventions

### Swift style
- 4-space indentation, no tabs.
- Trailing commas allowed in multi-line literals.
- `// MARK: -` to section files.
- Prefer `struct` over `class` unless reference semantics or SwiftData require otherwise.
- Use Swift Concurrency (`async`/`await`, `actor`) — no `DispatchQueue` for new code.
- No force-unwraps (`!`) outside test code. Use `guard let` / `if let`.

### File naming
- One primary type per file. File name matches type name.
- Views suffixed `View` (e.g., `MedicationDetailView.swift`).
- ViewModels suffixed `ViewModel`.
- Services suffixed `Service` (e.g., `NotificationService.swift`).

### SwiftData
- All `@Model` classes live in `Doseify/Models/`.
- Schema migrations: when you change a `@Model`, add a `VersionedSchema` and a migration plan. Never silently break existing user data.

### Testing
- Every Service gets unit tests.
- The TimezoneShiftEngine and DoseScorer must have comprehensive tests — these are the algorithmically tricky pieces.
- Use the Swift Testing framework (`@Test`, `#expect`) over XCTest where possible.
- No real network in tests. No real notifications in tests.

### Commit messages
- Imperative, concise: "Add TimezoneShiftEngine with east/west shortest-path logic".
- Reference SPEC section when relevant: "(see SPEC §2.4)".

## Hard rules

1. **No analytics, telemetry, crash reporting SDKs, or any third-party SDK that calls home.** Local-only.
2. **No silent timezone shifts.** Any change to dose timing must be triggered by a Trip the user created or a confirmation prompt after detection. Never re-anchor doses without user consent.
3. **Never delete or overwrite a logged dose event.** Only the user can edit one, and edits are stored as updates with the original preserved in history.
4. **`scheduledTimeHome` is sacred.** Store as UTC, never as a naive local Date. All timezone logic operates on `effectiveScheduledTime` derived from it.
5. **Do not use `!` force-unwraps in app code.** Only allowed in tests.
6. **Do not introduce dependencies without asking.** If a feature genuinely needs one (e.g., a charting library), stop and propose it.
7. **Critical Alerts entitlement requires Apple approval.** Build with `interruptionLevel = .timeSensitive` as the default; gate `.critical` behind a feature flag that's off until/unless the entitlement is granted.
8. **HealthKit medication writing is API-fragile.** Wrap it in a service that gracefully no-ops if the underlying API rejects. Do not let HealthKit failures block dose logging.

## When you're stuck or uncertain

- If the spec is ambiguous, stop and ask the user. Don't invent product behaviour.
- If an Apple API behaves differently than SPEC.md describes, flag it — SPEC was written without runtime verification of every API.
- If you're about to add a third-party dependency, stop and ask.
- If you're about to write more than ~200 lines without a test, stop and write the test first.

## Out of scope (do not build in v1)

- iCloud sync, CloudKit, multi-device.
- Ramp-up / taper dose schedules (data model should permit later, but no v1 UI).
- Family sharing, caregiver view, doctor portal.
- Drug interaction checking.
- Pharmacy integrations.
- Reading dose events from other apps as source of truth.

## User preferences

The owner prefers concise output and honest uncertainty. Avoid filler. When you're not sure something will work, say so rather than asserting confidence.
